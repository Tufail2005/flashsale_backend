📌 Project Overview

Flash sales create a "thundering herd" problem where thousands of users attempt to purchase a highly limited item at the exact same millisecond. Traditional REST APIs backed by relational databases frequently crash or heavily oversell inventory under this intense pressure.

The primary goal of this project was to decouple edge operations from asynchronous processing to build a serverless system capable of absorbing massive, instantaneous traffic spikes with zero overselling and ultra-low latency.

🛠️ Tech Stack
Edge API: Cloudflare Workers + Hono

State & Locking (The Bouncer): Upstash Redis

Message Broker: Cloudflare Queues

Database (The Cash Register): Neon Serverless PostgreSQL

Load Testing: Grafana k6



🏗️ Architecture Breakdown

<img width="1443" height="650" alt="flashsale_architecture" src="https://github.com/user-attachments/assets/25f66f50-4b2d-4819-9579-b5b1ae5bc683" />


1: The Edge (Redis): When traffic hits the API, the Cloudflare Worker intercepts the request and queries Upstash Redis using atomic operations. If stock is available, it allows the user in; if stock is zero, it immediately returns a 400 Out of Stock.

2: The Queue: Once a user secures an item, their order payload is instantly pushed to a Cloudflare Queue, freeing up the edge worker.

3: The Database: A background queue consumer pulls batches from the queue, processes the payloads, and permanently writes the finalized orders to Neon PostgreSQL.

4: Dead Letter Queue (DLQ) Rescue: If the Neon database experiences a permanent failure during a write, the system catches the failed payload in a dedicated flash-dlq. The DLQ worker automatically restocks the item in Redis (redis.incr) and logs the telemetry, completely preventing "ghost orders" where inventory is locked in a void.

Engineered and executed aggressive load tests using k6 to simulate a real-world flash sale drop.

The Parameters: 10,000 total requests generated by 500 concurrent Virtual Users (VUs) firing simultaneously for 50 available items.

The Results: Achieved 0 server crashes, 0 database locks, and 0 overselling. Exactly 50 users successfully secured the item, while the edge safely rejected the remaining 9,950 requests in single-digit milliseconds.


🚧 Challenges Faced & Learnings

The Noisy Neighbor & Event Loop Starvation: During peak traffic spikes, querying the database for the live admin dashboard caused the single-threaded Cloudflare Worker to hang, starving the event loop.

The Axiom Solution: Overcame this bottleneck by integrating Axiom for enterprise observability. Instead of optimizing the dashboard's polling logic or splitting workers, I used Cloudflare's ctx.executionCtx.waitUntil() to stream checkout telemetry to Axiom in the background. This "fire-and-forget" approach completely offloaded the analytics pipeline, reducing checkout latency to 0ms and preventing UI hangs.

Distributed Consistency: Successfully managed distributed state synchronization between the Redis edge cache (front door) and the PostgreSQL database (source of truth), ensuring failed DB writes roll back Redis inventory via the DLQ.


🔮 Future Scope

Implement multi-region Redis replication to lower global edge latency for international flash sales.

Add an automated notification worker (email/SMS) triggered by successful Neon DB insertions.

Abstract the Redis atomic locking operations into a reusable npm package for standard e-commerce campaigns.

### 💻 Local Setup & Development

Want to run the 10,000-user load test on your own machine? Here is how to run the worker and fire the k6 cannons.

**1. Clone the repository and install dependencies:**

git clone [https://github.com/yourusername/flash-sale-backend.git](https://github.com/yourusername/flash-sale-backend.git)
cd flash-sale-backend
npm install 

**2. Create .env :**

DATABASE_URL="postgresql://your-neon-db-url"
UPSTASH_REDIS_REST_URL="https://your-upstash-url"
UPSTASH_REDIS_REST_TOKEN="your-upstash-token"
JWT_SECRET="your-secret-key"
AXIOM_TOKEN="your-axiom-token"

**3 Run the Cloudflare Worker locally:**

npm run dev
# The worker will start on http://localhost:8787

**4. Unleash the Load Test:**

k6 run attack.js
